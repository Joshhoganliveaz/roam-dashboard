<decisions>
## Implementation Decisions

### Underwater / Negative Appreciation
- Show real numbers honestly — don't hide negative equity or skip sections
- Wrap negative values in SB7 reframing narrative: markets move in cycles, positioned for rebound, guide tone
- Display as '-$23,000' or '$23,000 below purchase price' with encouraging context — not false optimism, but not discouraging
- All sections still render for underwater homes; narrative tone adapts

### Ownership Duration
- Minimum practical ownership duration is ~12 months — Josh won't create pages for very recent purchases
- No need to design special handling for purchases under 3 months
- Earnings-per-week and appreciation numbers are meaningful at 1+ year of ownership

### Cash Purchase Handling
- Cash purchases are rare in Josh's client base — don't prioritize special handling
- Current defaults (zero mortgage) should work acceptably
- Claude's discretion on any minor display adjustments if zero mortgage is detected

### Large Number Display
- Always show full numbers — '$2,147,000' not '$2.1M'
- Homeowners want to see every digit; it feels more impressive and real
- Responsive text sizing (text-4xl/5xl/6xl) already handles overflow on mobile

### S&P 500 Comparison Data
- Replace fixed ~10% historical average with real S&P 500 monthly closing prices
- Use a static JSON data file bundled with the app (monthly granularity going back 30+ years)
- Update the file manually every few months — zero API dependency, works offline
- Lookup: find S&P 500 closing price on purchase month and current month, calculate actual return
- More credible comparison: "Your home gained $80K vs $27K if you'd invested in the S&P 500"

### Net Proceeds & Assumptions
- Standard defaults for closing costs and commission rates — not configurable per homeowner
- Net proceeds is understood as an estimate; homeowners know it's approximate

### Value Estimate Language
- Use subtle estimate language: "estimated current value" or "your home is worth approximately"
- Protects Josh legally without undermining confidence
- Hero numbers still present boldly — the qualifier is in surrounding narrative text, not the number itself

### Admin Preview & Validation
- Preview before publish: after saving, Josh sees a preview of the full homeowner page with a 'Publish' button
- Draft pages only visible to admin (existing draft/published state from Phase 1)
- Basic sanity checks on admin form inputs: positive numbers, reasonable range ($10K–$50M for values)
- Show warnings but still allow save — Josh knows best, validation is a safety net not a gate

### Number Formatting Standards
- Percentages: whole numbers only — '24%' not '24.3%' — cleaner for narrative tone
- Currency: never show cents — '$425,000' not '$425,000.00' — these are estimates, cents imply false precision
- Dates: conversational format — 'March 2023' for purchase dates, 'Updated March 12, 2026' for timestamps
- Negative currency: show with minus sign and reframing context

### Claude's Discretion
- Exact SB7 copy for underwater/negative scenarios
- How to handle sections when optional fields (rent estimate, loan details) use smart defaults vs were explicitly entered
- Minor cash purchase display adjustments
- Sanity check thresholds and warning copy
- Preview page layout and publish button placement
- S&P data file format and update cadence

</decisions>

- `src/lib/calculations.ts`: 13 pure functions + `computeHomeownerMetrics` orchestrator — returns 0 for NaN/Infinity edge cases, needs audit for underwater narrative support
- `src/lib/calculations.test.ts`: 41 unit tests including zero months, cash purchase, negative appreciation — extend with new edge cases
- `src/lib/format.ts`: `formatCurrency` and `formatPercent` utilities — audit for cents, decimal places, negative number handling
- `src/lib/defaults.ts`: `applyDefaults()` fills smart defaults for optional fields — relevant for missing data behavior
- `src/components/homeowner/*.tsx`: Six SB7 data section components — each needs edge case handling for negative/unusual values

- Pure calculation functions with no side effects — easy to add validation and edge case handling
- SB7 color coding: canyon for narrative, charcoal for hero numbers, charcoal/70 for supporting copy
- Intl.NumberFormat for currency/percent formatting — centralized in format.ts
- Server component fetches data, passes to client components — preview can reuse this pipeline
