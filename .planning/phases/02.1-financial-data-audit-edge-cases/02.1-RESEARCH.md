# Phase 02.1: Financial Data Audit & Edge Cases - Research

**Researched:** 2026-03-12
**Domain:** Financial calculation hardening, number formatting, edge case handling, admin workflow
**Confidence:** HIGH

## Summary

This phase hardens the existing financial data pipeline and display components built in Phases 1 and 2. The codebase is well-structured with 13 pure calculation functions, centralized formatting utilities, and six SB7 narrative section components. The primary work falls into four categories: (1) adapting all display components to handle negative/underwater scenarios with SB7 reframing narrative, (2) fixing number formatting to match Josh's standards (no cents, whole-number percentages, conversational dates), (3) replacing the fixed 10.5% S&P 500 average with real historical monthly data from a static JSON file, and (4) adding admin input validation with a preview-before-publish workflow.

The calculation engine already returns correct values for edge cases (negative appreciation, zero loan, zero months) and guards against NaN/Infinity. The gaps are in the display layer -- components use static positive-tone copy that breaks narratively for underwater homes -- and in formatting defaults (`formatPercent` uses 1 decimal by default, not whole numbers). The S&P 500 comparison currently uses `DEFAULTS.sp500AnnualReturn = 0.105` which needs to be replaced with a lookup against real monthly closing price data.

**Primary recommendation:** Work in three waves -- (1) formatting fixes and component edge case handling with SB7 narrative adaptation, (2) S&P 500 real data integration and admin validation, (3) preview-before-publish workflow. All changes are testable against the existing Vitest infrastructure.

### Pattern 2: S&P 500 Static Data Lookup
**What:** A static JSON file with monthly S&P 500 closing prices. A pure function looks up the price at purchase month and current month, calculating actual return.
**When to use:** Replacing the fixed 10.5% assumption in calcSP500Comparison.
**Example:**
```typescript
// src/lib/sp500-data.json structure
// Array of { date: "YYYY-MM", close: number }
// Monthly granularity, going back to ~1993 (30+ years)

// src/lib/sp500-lookup.ts
import sp500Data from "./sp500-data.json";

interface SP500DataPoint {
  date: string; // "YYYY-MM"
  close: number;
}

/**
 * Look up S&P 500 closing price for a given month.
 * Falls back to nearest available month if exact match not found.
 */
export function getSP500Price(yearMonth: string): number | null {
  const point = (sp500Data as SP500DataPoint[]).find(d => d.date === yearMonth);
  if (point) return point.close;
  // Fallback: find nearest month
  const sorted = (sp500Data as SP500DataPoint[])
    .filter(d => d.date <= yearMonth)
    .sort((a, b) => b.date.localeCompare(a.date));
  return sorted[0]?.close ?? null;
}

/**
 * Calculate actual S&P 500 return over ownership period.
 * If lookup fails, falls back to historical average.
 */
export function calcRealSP500Return(
  downPayment: number,
  purchaseDate: Date,
  currentDate: Date = new Date()
): { sp500Return: number; isRealData: boolean } {
  const purchaseMonth = `${purchaseDate.getFullYear()}-${String(purchaseDate.getMonth() + 1).padStart(2, "0")}`;
  const currentMonth = `${currentDate.getFullYear()}-${String(currentDate.getMonth() + 1).padStart(2, "0")}`;

  const purchasePrice = getSP500Price(purchaseMonth);
  const currentPrice = getSP500Price(currentMonth);

  if (purchasePrice && currentPrice) {
    const sp500GrowthRate = currentPrice / purchasePrice;
    return {
      sp500Return: downPayment * sp500GrowthRate - downPayment,
      isRealData: true,
    };
  }

  // Fallback to historical average
  const months = (currentDate.getTime() - purchaseDate.getTime()) / (30.44 * 24 * 60 * 60 * 1000);
  const years = months / 12;
  return {
    sp500Return: downPayment * Math.pow(1.105, years) - downPayment,
    isRealData: false,
  };
}
```

### Pattern 3: Admin Validation Warnings (Non-Blocking)
**What:** Validation functions return warnings that display in the form but do not prevent save.
**When to use:** Admin form input validation.
**Example:**
```typescript
// src/lib/validation.ts
interface ValidationWarning {
  field: string;
  message: string;
}

export function validateHomeownerInputs(store: {
  purchasePrice: number;
  currentValue: number;
  loanAmount: number | null;
  interestRate: number | null;
}): ValidationWarning[] {
  const warnings: ValidationWarning[] = [];

  if (store.purchasePrice < 10_000 || store.purchasePrice > 50_000_000) {
    warnings.push({
      field: "purchasePrice",
      message: `$${store.purchasePrice.toLocaleString()} seems unusual. Expected range: $10K - $50M.`,
    });
  }

  if (store.currentValue < 10_000 || store.currentValue > 50_000_000) {
    warnings.push({
      field: "currentValue",
      message: `$${store.currentValue.toLocaleString()} seems unusual. Expected range: $10K - $50M.`,
    });
  }

  if (store.interestRate !== null && (store.interestRate < 0 || store.interestRate > 0.15)) {
    warnings.push({
      field: "interestRate",
      message: `${(store.interestRate * 100).toFixed(1)}% seems unusual. Expected range: 0% - 15%.`,
    });
  }

  return warnings;
}
```

### Anti-Patterns to Avoid
- **Hiding sections for underwater homes:** User explicitly wants all sections to render. Never conditionally skip rendering a section based on negative values.
- **Showing cents on currency:** User wants clean whole numbers. Never pass `decimals: 2` to formatCurrency for display values.
- **Abbreviating large numbers:** User wants full digits. Never format as "$2.1M".
- **Blocking save on validation warnings:** Warnings are advisory. Josh knows best -- validation is a safety net not a gate.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Currency formatting | Custom regex/string manipulation | Intl.NumberFormat via existing formatCurrency | Handles locale, negatives, commas correctly |
| Date formatting | Manual month name arrays | date-fns `format()` with 'MMMM yyyy' | Already in project, handles all locales |
| S&P 500 data API | Real-time API integration | Static JSON file updated manually | Zero runtime dependency, works offline, user's explicit decision |
| Form validation library | Zod/Yup schema | Simple pure function returning warnings array | Overkill for 5-6 field checks, non-blocking pattern doesn't need schema |

## Common Pitfalls

### Pitfall 1: formatPercent Default Decimal Places
**What goes wrong:** Current `formatPercent` defaults to 1 decimal place (`decimals = 1`), producing "24.3%" instead of the required "24%".
**Why it happens:** The function was written before the formatting standard was established.
**How to avoid:** Change the default parameter from `decimals = 1` to `decimals = 0`. Audit all existing call sites to ensure none rely on the decimal.
**Warning signs:** Any percentage on the page showing a decimal point.

### Pitfall 2: Intl.NumberFormat Negative Currency
**What goes wrong:** `formatCurrency(-23000)` produces `-$23,000` which is correct per the user's decision. No pitfall here -- verify it works.
**Why it matters:** Some locales format negatives as `($23,000)` in accounting format. The `en-US` currency format uses the minus sign, which matches the requirement.
**How to avoid:** No change needed -- just verify the existing behavior in tests.

### Pitfall 3: PurchaseStory Date Format
**What goes wrong:** Currently formats as "January 15, 2022" (MMMM d, yyyy). User wants conversational format "March 2023" for purchase dates.
**Why it happens:** The component uses date-fns `format(parseISO(purchaseDate), "MMMM d, yyyy")`.
**How to avoid:** Change to `format(parseISO(purchaseDate), "MMMM yyyy")` -- drop the day, keep month and year.

### Pitfall 4: S&P 500 JSON File Size
**What goes wrong:** 30+ years of monthly data (360+ rows) in JSON is tiny (~15KB), but could grow if someone adds daily data by mistake.
**Why it happens:** Unclear documentation on granularity expectations.
**How to avoid:** Keep monthly granularity only. Document the format and update process in a comment at the top of the lookup module.

### Pitfall 5: Preview Route Must Use Admin Auth but Draft Data
**What goes wrong:** The existing `/h/[slug]/page.tsx` filters by `status: "published"`. A preview route needs to show draft pages to authenticated admins.
**Why it happens:** The public route correctly restricts to published. Preview needs a separate route that uses the admin Supabase client.
**How to avoid:** Create preview as an admin sub-route (`/admin/[id]/preview`) that uses the admin client to fetch by ID (not slug), skipping the status filter. This reuses the same `applyDefaults` + `computeHomeownerMetrics` + `HomeownerPage` pipeline.

### Pitfall 6: Negative Appreciation Chart
**What goes wrong:** The AppreciationChart generates data points using the annualized appreciation rate. If appreciation is negative, the curve correctly shows decline. However, the narrative copy says "Watch how your investment has grown" which is wrong for a declining home.
**Why it happens:** Static copy assumes positive growth.
**How to avoid:** Add conditional narrative copy to the chart section header, similar to other sections.

## Open Questions

1. **S&P 500 Data Source for Initial File**
   - What we know: Need monthly closing prices from ~1993 to present in JSON format
   - What's unclear: Best source to generate the initial file. Macrotrends and FRED both offer downloads, but format varies.
   - Recommendation: Use FRED (fred.stlouisfed.org) as the authoritative source. Download CSV, convert to JSON with a simple script. Document the process for Josh's future manual updates.

2. **S&P 500 "Current Month" Timing**
   - What we know: The file will be updated every few months. The most recent data point may be 1-3 months old.
   - What's unclear: Should we show "as of February 2026" when the data is slightly stale?
   - Recommendation: Use the most recent available month in the data file. The SP500Callout can optionally note "based on data through [last month in file]" if the gap exceeds 3 months.

3. **Admin Preview Auth Pattern**
   - What we know: Preview needs to be admin-only and show draft content
   - What's unclear: Whether to use middleware auth check or server-side cookie check
   - Recommendation: Place under `/admin/[id]/preview` which is already behind auth middleware from Phase 1.
